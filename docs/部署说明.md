# OpenMall 部署说明

> 本文档用于说明 OpenMall 项目的运行环境、依赖安装、本地部署方式、基础服务器部署方式以及常见问题处理，适用于当前的 **Spring Boot 3.x + Spring Security + JWT + MyBatis-Plus + Knife4j + MySQL 8.0.34 + Redis + Spring Cache + Druid + RabbitMQ + 阿里云 OSS + Vue 3 + Nginx + Docker** 架构。

---

## 1. 文档目标

本部署说明主要解决以下问题：

- 如何在本地启动 OpenMall
- 如何准备 MySQL、Redis、RabbitMQ 环境
- 如何启动后端、用户端、商家端
- 如何在基础服务器环境中完成部署
- 如何定位常见启动与部署问题

## 2. 项目架构说明

OpenMall 当前采用前后端分离架构，包含以下部分：

- `openmall-server`：Spring Boot 后端服务
- `openmall-user-web`：用户端前端项目
- `openmall-merchant-web`：商家端前端项目
- `MySQL`：核心业务数据库
- `Redis`：缓存与登录态支持
- `RabbitMQ`：异步消息队列
- `阿里云 OSS`：文件与图片存储

## 5. 本地开发环境部署

### 5.1 第一步：准备数据库

确保本地已安装并启动 MySQL。

#### 5.1.1 创建数据库

```sql
CREATE DATABASE openmall DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
```

#### 5.1.2 导入初始化脚本

```bash
mysql -uroot -p openmall < sql/openmall.sql
```

如果后续项目拆分了初始化脚本，也可以分别执行：

```bash
mysql -uroot -p openmall < sql/init-schema.sql
mysql -uroot -p openmall < sql/init-data.sql
```

---

### 5.2 第二步：准备 Redis

启动 Redis 服务。

默认配置下可使用：

```text
host: 127.0.0.1
port: 6379
password: 无（本地开发环境可选）
```

如使用 Docker 启动，可参考：

```bash
docker run -d --name openmall-redis -p 6379:6379 redis:7 --requirepass "你的强密码"
```

---

### 5.3 第三步：准备 RabbitMQ

启动 RabbitMQ 服务。

默认配置下可使用：

```text
host: 127.0.0.1
port: 5672
management port: 15672
username: openmall
password: 自定义强密码
```

> 注意：RabbitMQ 默认 `guest` 用户通常只允许通过 localhost 访问。本地开发即使能用 `guest/guest`，也不建议写入部署文档；如果部署在服务器、容器或通过非 `127.0.0.1` 地址访问，请创建独立用户并按最小权限原则分配权限。

如使用 Docker 启动，可参考：

```bash
docker run -d --name openmall-rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management
```

RabbitMQ 管理后台访问地址：

```text
http://localhost:15672
```

---

### 5.4 第四步：配置后端项目

后端配置文件通常位于：

```text
openmall-server/src/main/resources/application.yml
```

建议配置项包括：

- 服务端口
- 数据库连接
- Redis 连接
- RabbitMQ 连接
- JWT 配置
- Druid 连接池配置
- Knife4j 接口文档配置
- 阿里云 OSS 配置

示例配置参考：

```yaml
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://127.0.0.1:3306/openmall?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver
  data:
    redis:
      host: 127.0.0.1
      port: 6379
  rabbitmq:
    host: 127.0.0.1
    port: 5672
    username: ${RABBITMQ_USERNAME:openmall}
    password: ${RABBITMQ_PASSWORD:change_me}

  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 20MB

springdoc:
  api-docs:
    enabled: true

knife4j:
  enable: true
```

> 注意：实际字段名称可根据 Spring Boot 版本与项目实现方式微调。

如果项目接入阿里云 OSS，建议补充：

```yaml
aliyun:
  oss:
    endpoint: ${OSS_ENDPOINT:oss-cn-shanghai.aliyuncs.com}
    access-key-id: ${OSS_ACCESS_KEY_ID:}
    access-key-secret: ${OSS_ACCESS_KEY_SECRET:}
    bucket-name: ${OSS_BUCKET_NAME:openmall-bucket}
```

> 强烈建议：不要把真实 `AccessKey ID / Secret` 直接写入 `application.yml` 并提交到 Git 仓库。生产环境优先通过环境变量、服务器配置或云平台凭据管理方式注入。

### 5.5 第五步：启动后端服务

进入后端目录：

```bash
cd openmall-server
```

使用 Maven 启动：

```bash
mvn spring-boot:run
```

或先打包后运行：

```bash
mvn clean package -DskipTests
java -jar target/openmall-server.jar
```

启动成功后，可访问：

```text
http://localhost:8080
```

如果项目后续接入 Swagger / Knife4j，也可访问接口文档页面。

开发环境中建议通过以下地址访问 Knife4j：

```text
http://localhost:8080/doc.html
```

---

### 5.6 第六步：启动用户端前端

进入用户端目录：

```bash
cd openmall-user-web
```

安装依赖：

```bash
npm install
```

启动开发环境：

```bash
npm run dev
```

默认访问地址通常为：

```text
http://localhost:5173
```

### 5.6.1 关于本地开发跨域（CORS）

当前本地开发环境中：

- 后端接口：`http://localhost:8080`
- 用户端前端：`http://localhost:5173`
- 商家端前端：`http://localhost:5174`

由于端口不同，这三者属于不同 Origin。若后端未正确配置 CORS，前端直接请求后端接口时会出现浏览器跨域错误。

建议做法：

1. **开发环境后端显式配置 CORS 白名单**，仅允许：
   - `http://localhost:5173`
   - `http://localhost:5174`
2. 不要使用 `*` 作为生产环境允许来源
3. 如项目接入 Spring Security，应确保安全配置中也正确放行跨域预检请求

建议说明写法：

```text
开发环境请在后端显式允许来自 http://localhost:5173 和 http://localhost:5174 的跨域请求。
生产环境请仅允许实际前端域名，避免使用 *。
```

## 6. 前端环境变量建议

建议用户端与商家端分别维护环境变量文件。

### 6.1 用户端示例

文件：`.env.development`

```env
VITE_APP_TITLE=OpenMall 用户端
VITE_API_BASE_URL=/api/v1
VITE_OSS_BASE_URL=https://your-bucket.oss-cn-shanghai.aliyuncs.com
```

说明：

- 若前端开发环境通过 Vite 代理后端接口，推荐使用相对路径 `/api/v1`
- 若不使用代理，则必须保证后端已配置 CORS，前端才可直接请求 `http://localhost:8080`

可选的 Vite 代理思路：

```text
开发环境可通过 Vite dev server 将 /api 代理到 http://localhost:8080，减少本地联调时的跨域问题。
```

### 6.2 商家端示例

文件：`.env.development`

```env
VITE_APP_TITLE=OpenMall 商家端
VITE_API_BASE_URL=/merchant/api/v1
VITE_OSS_BASE_URL=https://your-bucket.oss-cn-shanghai.aliyuncs.com
```

说明：

- 若前端开发环境通过 Vite 代理后端接口，推荐使用相对路径 `/merchant/api/v1`
- 若不使用代理，则必须保证后端已配置 CORS

## 6.3 OSS 部署说明

如果项目接入阿里云 OSS，部署前需要准备：

- OSS Bucket
- AccessKey ID
- AccessKey Secret
- Bucket 对应地域 Endpoint

建议：

- 开发环境使用单独测试 Bucket
- 生产环境将密钥写入环境变量或服务器配置，不直接写入代码仓库
- 商品图片、店铺资源、富文本资源应区分业务目录前缀

推荐环境变量示例：

```bash
export OSS_ENDPOINT=oss-cn-shanghai.aliyuncs.com
export OSS_ACCESS_KEY_ID=请在服务器环境中注入真实值
export OSS_ACCESS_KEY_SECRET=请在服务器环境中注入真实值
export OSS_BUCKET_NAME=openmall-bucket
```

## 7. 基础服务器部署说明

本节适用于单机 Linux 服务器部署，适合作为学习项目或演示环境。

### 7.1 服务器建议配置

- 2 核 CPU
- 4GB 内存
- 40GB 以上磁盘
- Linux（Ubuntu / CentOS）

### 7.2 服务器安装依赖

部署前建议准备：

- JDK 17
- Maven（如需服务器打包）
- MySQL
- Redis
- RabbitMQ
- Nginx
- Docker

### 7.3 初始化服务器数据库

在启动后端服务之前，必须先在服务器上完成数据库初始化。

#### 7.3.1 创建数据库

```sql
CREATE DATABASE openmall DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
```

#### 7.3.2 导入初始化脚本

```bash
mysql -uroot -p openmall < /opt/openmall/sql/openmall.sql
```

说明：

- 若使用独立的初始化脚本，请按顺序导入 schema 与 data
- 建议在测试环境和生产/演示环境使用独立数据库账号

### 7.4 后端部署步骤

#### 7.4.1 打包后端

在本地或服务器执行：

```bash
cd openmall-server
mvn clean package -DskipTests
```

生成 Jar 包后上传至服务器，例如：

```text
/opt/openmall/openmall-server.jar
```

#### 7.4.2 启动后端

```bash
java -jar /opt/openmall/openmall-server.jar --spring.profiles.active=prod > /dev/null 2>&1 &
```
建议在后端的 logback-spring.xml 或 application.yml 中配置日志切分策略，避免单个日志文件过大占用磁盘。
如需后台运行：

```bash
nohup java -jar /opt/openmall/openmall-server.jar --spring.profiles.active=prod > /opt/openmall/logs/server.log 2>&1 &
```

### 7.5 前端部署步骤

#### 7.5.1 用户端打包

```bash
cd openmall-user-web
npm install
npm run build
```

打包输出目录通常为：

```text
dist/
```

#### 7.5.2 商家端打包

```bash
cd openmall-merchant-web
npm install
npm run build
```

同样会生成 `dist/` 目录。

### 7.6 Nginx 部署示例

可将两个前端项目分别部署为不同访问路径或不同二级域名。

#### 示例一：不同路径部署

- 用户端：`/`
- 商家端：`/merchant/`

示例配置：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        root /opt/openmall/user-web/dist;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    location /merchant/ {
        alias /opt/openmall/merchant-web/dist/;
        index index.html;
        try_files $uri $uri/ /merchant/index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /merchant/api/ {
        proxy_pass http://127.0.0.1:8080/merchant/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

> 实际 Nginx 配置应根据前端路由模式与部署路径再做适配。

## 9. 常见问题排查

### 9.1 后端启动失败

优先检查：

- JDK 版本是否正确
- `application.yml` 是否配置正确
- MySQL 是否启动
- Redis 是否启动
- RabbitMQ 是否启动
- Druid 配置是否正确
- OSS 配置是否缺失
- 数据库连接用户名密码是否正确

### 9.2 数据库连接失败

检查：

- 数据库服务是否正常运行
- 数据库是否已创建
- 端口是否正确
- 用户权限是否足够

### 9.3 前端接口请求失败

检查：

- 后端服务是否启动
- 前端环境变量中的 API 地址是否正确
- 是否存在跨域问题
- 后端是否已正确配置允许 `http://localhost:5173`、`http://localhost:5174` 的 CORS 白名单
- Token 是否已正确携带
- Nginx 反向代理路径是否正确

### 9.4 RabbitMQ 无法连接

检查：

- 服务端口是否开放
- 是否仍在使用默认 `guest` 账号进行远程连接
- 自定义 RabbitMQ 用户是否已正确创建并授权
- 管理台是否正常登录

### 9.5 Redis 无法连接

检查：

- Redis 服务是否正常启动
- 端口与密码配置是否一致
- 防火墙是否拦截连接

### 9.6 OSS 文件上传失败

检查：

- AccessKey 是否正确
- Bucket 是否存在
- Endpoint 是否与 Bucket 地域一致
- 服务端是否正确加载 OSS 配置
- 前端上传文件大小与类型是否超限

## 10. 部署环境建议

### 10.1 开发环境

适用于本地开发与调试：

## 11. 部署顺序建议

推荐部署顺序如下：

1. 启动 MySQL
2. 导入数据库脚本
3. 启动 Redis
4. 启动 RabbitMQ
5. 配置并启动后端
6. 打包并部署用户端前端
7. 打包并部署商家端前端
8. 配置 Nginx 反向代理
9. 配置 OSS 与文件上传能力
10. 验证用户端与商家端核心流程

部署完成后，建议至少验证以下内容：

- 库存修改是否正常
- 订单列表与详情是否正常
- 发货操作是否正常

### 12.3 后端验证

- 服务是否可正常访问
- 接口是否返回统一格式
- 日志是否正常输出
- 数据库写入是否正常

## 13. 总结

OpenMall 当前阶段适合采用“本地开发 + 单机部署”的方式运行。对于学习项目而言，最重要的是先完成一套稳定可运行的用户端、商家端与后端联动流程。

在此基础上，后续可进一步扩展：

- Docker 化部署
- CI/CD 自动部署
- HTTPS 与域名配置
- 多环境配置管理
- 日志监控与告警能力
