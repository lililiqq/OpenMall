# OpenMall 项目开发规范

> 本文档用于统一 OpenMall 项目的开发规范，适用于后端、用户端前端、商家端前端的日常开发、联调、测试与文档维护。

---

## 1. 文档目的

本规范文档用于解决以下问题：

- 保持项目目录结构清晰
- 统一命名风格与代码风格
- 减少多人协作时的沟通成本
- 提高代码可读性与可维护性
- 保证用户端、商家端、后端三方协作一致

---

## 2. 项目整体规范

### 2.1 项目定位

OpenMall 是一个基于 **Spring Boot + Vue3** 的前后端分离模拟淘宝项目，包含：

- 用户端
- 商家端
- 后端服务

当前阶段采用 **模块化单体架构**，开发重点是保证核心业务闭环，而不是追求过度复杂的技术方案。

### 2.2 开发基本原则

开发过程中应遵循以下原则：

1. **先设计，后编码**
2. **先文档，后接口联调**
3. **小步提交，避免一次性大改**
4. **优先保证业务正确性，再考虑性能优化**
5. **遵循统一命名与目录规范**
6. **用户端、商家端、后端字段命名保持一致**

- `upgrade-v1.1.sql`：版本升级脚本

## 4. 后端开发规范

### 4.1 包结构规范

后端建议按模块划分目录，不建议所有代码堆在一起。

建议结构：

```text
com.openmall
├─ common
├─ config
├─ modules
│  ├─ user
│  ├─ merchant
│  ├─ shop
│  ├─ address
│  ├─ category
│  ├─ product
│  ├─ cart
│  ├─ order
│  ├─ inventory
│  ├─ payment
│  └─ merchant-center
```

每个模块内建议进一步拆分：

- `controller`
- `service`
- `service.impl`
- `entity`
- `dto`
- `vo`
- `mapper` 或 `repository`
- `enums`

说明：

- 持久层框架统一使用 **MyBatis-Plus**
- 数据库连接池统一使用 **Druid**
- 接口文档统一接入 **Knife4j**
- 安全认证统一使用 **Spring Security + JWT**

### 4.2 命名规范

#### 类命名

- Controller 类：`UserController`
- Service 接口：`UserService`
- Service 实现类：`UserServiceImpl`
- Entity 类：`UserEntity` 或 `User`
- DTO 类：`CreateOrderDTO`
- VO 类：`OrderDetailVO`
- 枚举类：`OrderStatusEnum`

#### 方法命名

方法名要体现业务语义，例如：

- `registerUser`
- `loginUser`
- `createOrder`
- `cancelOrder`
- `updateProductStatus`

不要使用无业务意义的命名，例如：

- `doSave`
- `handle`
- `processData`

#### 金额字段命名与类型规范

- 后端涉及金额的 Java 字段统一使用 `BigDecimal`
- 数据库金额字段统一使用 `decimal` 类型
- 项目必须统一金额单位，禁止部分模块用“元”、部分模块用“分”混用
- 若当前项目采用“元”为单位，建议统一字段命名为：
  - `total_amount`
  - `pay_amount`
  - `freight_amount`
- 若后端字段为金额对象，推荐命名：
  - `totalAmount`
  - `payAmount`

#### 时间字段命名与类型规范

- Java 时间字段统一优先使用 `LocalDateTime`
- 创建/更新时间统一使用：
  - `createTime`
  - `updateTime`
- 表示业务事件发生时间时，字段名称应体现语义，例如：
  - `payTime`
  - `cancelTime`
  - `deliveryTime`
  - `finishTime`
- 不要使用语义模糊的时间字段名，例如：
  - `time`
  - `date`
  - `handlerTime`

#### 布尔字段命名规范

- Java 布尔属性命名必须语义清晰，例如：
  - `enabled`
  - `deleted`
  - `defaulted`
  - `checked`
- 数据库字段建议避免使用 `is_xxx` 命名，减少跨框架映射、序列化与反序列化歧义
- 需要表达状态时，优先使用有业务语义的字段名，而不是模糊布尔标记

### 4.3 接口层规范

- Controller 仅负责接收参数与返回结果
- 不要在 Controller 中直接编写复杂业务逻辑
- 接口返回统一响应结构
- 接口路径命名保持 RESTful 风格

示例：

- `GET /api/v1/products`
- `POST /api/v1/orders`
- `PUT /merchant/api/v1/products/{id}`

### 4.4 Service 层规范

- Service 层负责核心业务逻辑
- 一个方法尽量只完成一个明确业务动作
- 涉及事务的方法应明确事务边界
- 不要在 Service 中写与当前模块无关的横向逻辑

### 4.5 DTO / VO 规范

- 请求对象统一使用 DTO
- 返回对象统一使用 VO
- 不建议直接将数据库实体暴露给前端
- DTO 与 VO 字段应尽量贴近业务语义

### 4.6 异常处理规范

- 统一使用全局异常处理
- 自定义业务异常类型
- 错误信息应尽量明确，不要返回模糊提示
- 必须提供统一的 `GlobalExceptionHandler`
- 禁止在业务代码中到处散落 `try-catch` 后吞异常
- 业务异常应抛出统一的业务异常类型，由全局异常处理器统一转换为标准响应

建议统一处理以下错误：

- 参数校验失败
- 资源不存在
- 用户未登录
- 权限不足
- 商品已下架
- 库存不足
- 订单状态异常

建议基础模板统一放在后端公共包中，例如：

- `common/api/Result.java`
- `common/api/ErrorCode.java`
- `common/exception/BusinessException.java`
- `common/exception/GlobalExceptionHandler.java`

### 4.6.1 状态码与统一返回规范

- 所有接口必须统一返回标准结构，例如：`Result<T>`
- HTTP 状态码遵循基础语义，业务错误码遵循项目统一约定
- 不允许同类错误在不同接口中返回完全不同的业务码

建议最少统一以下类型：

- 成功：`0`
- 参数错误：`4001`
- 未登录或令牌无效：`4002`
- 无权限访问：`4003`
- 资源不存在：`4004`
- 状态不允许当前操作：`4005`
- 库存不足：`4006`
- 通用业务失败：`4000`
- 系统异常：`5000`

### 4.7 数据库操作规范

- 不允许在业务代码中拼接原生 SQL 字符串
- 更新与删除操作必须带明确条件
- 涉及库存、订单、支付状态时应格外注意事务边界
- 批量操作前先确认影响范围

补充要求：

- MyBatis-Plus 适用于基础 CRUD、分页、条件查询
- 复杂查询可以保留自定义 SQL，但必须保证可维护性
- Druid 连接池配置需区分开发环境与生产环境
- SQL 日志输出策略应避免在生产环境中泄露敏感信息
- MyBatis 参数绑定统一优先使用 `#{}`
- 禁止在 XML 或注解 SQL 中直接使用 `${}` 拼接用户输入
- 若排序字段、表名等极特殊场景需要动态拼接，必须经过白名单校验后再处理

### 4.8 日志规范

- 核心业务操作需记录日志
- 错误日志必须包含上下文信息
- 禁止打印密码、Token、身份证号等敏感信息

建议记录的日志包括：

- 用户登录
- 下单
- 支付
- 订单取消
- 商家发货
- 异常报错

### 4.9 接口文档规范

- 接口字段级、参数级、调试级事实来源应以 **Knife4j / OpenAPI** 文档为准
- `docs/API.md` 主要用于维护宏观接口分组、核心流程、联调说明与错误码约定
- 所有接口变更必须同步更新在线接口文档注解
- 用户端接口与商家端接口必须明确分组
- 请求参数、返回示例、错误码说明应与文档保持一致

### 4.10 缓存与消息队列规范

- Redis 缓存统一通过 **Spring Cache** 或统一缓存工具层接入
- 不允许在业务代码中无约束地散落缓存 Key
- 缓存 Key 命名建议带业务前缀，例如：`product:detail:{id}`
- RabbitMQ 消费逻辑必须考虑幂等处理
- 消息队列仅用于明确需要异步化的流程，不应滥用

### 4.11 文件存储规范

- 图片、附件等文件统一走后端上传接口
- 实际文件统一存储到 **阿里云 OSS**
- 数据库中可记录文件元信息，如业务类型、URL、对象 Key、上传人
- 文件路径建议按业务分目录，例如：`product/`、`shop/`、`editor/`

## 5. 前端开发规范

### 5.1 前端项目划分

前端分为两个独立项目：

- `openmall-user-web`：用户端
- `openmall-merchant-web`：商家端

两端项目可共享开发思想，但应在业务、页面、权限上保持隔离。

### 5.2 目录结构建议

以 Vue3 项目为例：

```text
src/
├─ api/
├─ assets/
├─ components/
├─ layouts/
├─ router/
├─ stores/
├─ styles/
├─ utils/
├─ views/
└─ App.vue
```

### 5.3 页面命名规范

- 页面目录使用小写英文或短横线命名
- 组件文件建议使用 PascalCase

示例：

- `views/product-list/index.vue`
- `views/order-detail/index.vue`
- `components/ProductCard.vue`

### 5.4 API 调用规范

- 所有接口统一放在 `api/` 目录中管理
- 不允许在页面组件中直接写 Axios 请求地址
- 接口命名应与业务一致

示例：

- `getProductList`
- `getProductDetail`
- `createOrder`
- `merchantLogin`

补充要求：

- 用户端与商家端接口目录应分开维护
- 环境变量中统一维护 API Base URL
- 不要在组件中硬编码 OSS 域名、接口地址、文件上传路径

### 5.5 状态管理规范

- 用户登录态使用 Pinia 管理
- 商家登录态与用户登录态分开管理
- 不同业务模块状态尽量分 store 管理，避免单个 store 过于臃肿

### 5.6 页面开发规范

- 用户端页面以购物流程为中心
- 商家端页面以管理效率为中心
- 页面应明确区分加载中、空数据、错误状态
- 表单提交前必须做前端基础校验

### 5.7 文件上传与静态资源规范

- 商品图片、店铺图片、富文本图片上传需统一走后端上传接口
- 图片地址来源应统一为 OSS 返回地址或后端转换后的可访问地址
- 上传前端需要校验文件类型、大小、数量等基本规则

## 6. 接口联调规范

### 6.1 联调前要求

在联调前必须先确认：

- 接口文档已更新
- 请求参数和返回字段已明确
- 状态码与错误信息约定一致
- 鉴权方式已统一

### 6.2 联调过程要求

- 前端发现字段变更时，先更新文档再修改代码
- 后端变更接口时，需同步更新 `docs/API.md`
- 联调问题优先检查请求参数、Header、Token、状态流转
- 联调期间建议通过 Knife4j 统一测试用户端与商家端接口

### 6.3 Mock 规范

在后端接口未完成时，可先使用 Mock 数据开发页面，但需遵守：

- Mock 数据字段名称必须与接口文档一致
- Mock 数据结构尽量贴近真实业务场景
- 联调完成后及时清理无用 Mock 数据

## 7. 数据库设计与变更规范

### 7.1 建表规范

- 表名使用小写下划线风格
- 主键统一使用 `id`
- 时间字段统一使用：
  - `create_time`
  - `update_time`
- 状态字段命名统一，如：
  - `status`
  - `order_status`
  - `pay_status`

### 7.2 字段命名规范

- 字段名使用小写下划线
- 见名知意，避免缩写过度

示例：

- `receiver_name`
- `detail_address`
- `shop_id`
- `product_id`

### 7.3 数据变更规范

- 修改表结构前必须保留变更脚本
- 禁止直接在线上或共享数据库手工改表而不留记录
- 重大字段变更必须同步更新设计文档与接口文档

## 8. Git 提交规范

### 8.1 提交原则

- 一次提交只做一类事情
- 提交信息应清晰表达本次变更内容
- 不要把无关文件混在同一次提交中
- 提交信息统一采用 **Conventional Commits / Angular 风格**

### 8.2 提交信息建议格式

```text
feat(order): 新增用户下单接口
fix(cart): 修复购物车数量更新异常
docs(api): 补充 API 接口文档
refactor(order): 重构订单创建流程
style(frontend): 统一前端页面样式
test(order): 增加订单模块测试用例
```

补充要求：

- 推荐格式：`type(scope): subject`
- `subject` 应简洁描述改动目的，避免空泛语句
- 常用 `type`：`feat`、`fix`、`docs`、`refactor`、`style`、`test`、`chore`

### 8.3 分支建议

- `main`：主分支
- `dev`：开发分支
- `feature/xxx`：功能分支
- `fix/xxx`：修复分支
- `docs/xxx`：文档分支

## 9. 测试规范

### 9.1 后端测试

- 核心业务应至少覆盖基础测试
- 重点测试以下流程：
  - 用户登录
  - 加入购物车
  - 创建订单
  - 取消订单
  - 模拟支付
  - 商家发货

### 9.2 前端测试

- 页面需验证基础交互流程
- 表单项、按钮状态、接口异常提示需重点检查
- 用户端与商家端都要验证登录态与权限控制

### 9.3 联调测试

- 至少完成一次完整的用户下单链路测试
- 至少完成一次完整的商家订单处理链路测试

## 10. 文档维护规范

### 10.1 文档更新原则

出现以下情况时，必须同步更新文档：

- 新增模块
- 新增接口
- 修改接口字段
- 修改数据库结构
- 修改核心流程
- 修改 Knife4j/OpenAPI 注解

### 10.2 文档优先级

建议维护顺序：

1. `README.md`
2. `docs/DESIGN.md`
3. 在线接口文档（Knife4j / OpenAPI）
4. `docs/API.md`
5. `docs/技术栈说明.md`
6. `docs/README-开发规范.md`
7. `docs/部署说明.md`

### 10.3 文档风格要求

- 使用中文描述
- 标题层级清晰
- 示例尽量完整
- 避免只写结论不写上下文

## 11. 安全规范

### 11.1 基础安全要求

- 密码必须加密存储
- Token 不得明文记录到日志
- 禁止返回敏感字段给前端
- 用户端与商家端权限必须严格隔离

### 11.2 数据访问控制

- 用户只能访问自己的购物车、地址、订单
- 商家只能访问自己店铺的商品、订单、库存
- 严禁越权查询其他用户或其他商家的数据

### 11.3 商家端越权防护

- 商家端接口不得仅依赖前端传入的 `merchantId`、`shopId` 进行权限判断
- 后端必须从 JWT Token 中解析当前商家身份，再校验资源归属
- 商品、订单、库存、店铺信息等商家侧资源必须校验是否属于当前登录商家

### 11.4 SQL 注入防护

- 即使使用 MyBatis-Plus，也不得放松 SQL 注入防护要求
- XML 或注解 SQL 中优先使用 `#{}` 参数绑定
- 禁止直接拼接用户输入形成 SQL 片段

## 12. 性能与扩展规范

### 12.1 缓存规范

- 热点商品详情可使用 Redis 缓存
- 商品列表可根据访问频率决定是否缓存
- 缓存设计必须考虑失效策略与一致性问题
- 缓存统一优先接入 Spring Cache，避免不同模块各自维护缓存风格

### 12.2 MQ 使用规范

- RabbitMQ 优先用于异步任务
- 不要为使用 MQ 而使用 MQ
- 消费逻辑必须考虑幂等处理

### 12.3 部署与环境规范

- 本地开发环境可以优先使用 Docker 拉起 MySQL、Redis、RabbitMQ
- 前端发布默认通过 Nginx 托管静态资源
- 不同环境的配置应通过环境变量或配置文件区分，不应写死在代码中

### 12.4 扩展原则

- 新功能优先复用现有模块能力
- 避免出现跨模块强耦合调用
- 如果未来复杂度明显增加，再考虑微服务拆分

---

## 13. 当前阶段开发建议

建议按以下顺序推进开发：

1. 搭建后端基础工程
2. 完成数据库设计与建表脚本
3. 实现用户认证与商品模块
4. 实现购物车模块
5. 实现订单与支付模拟模块
6. 实现商家端登录、商品管理、订单管理
7. 补充 Redis 与 RabbitMQ 增强能力
8. 完善文档与测试

## 14. 总结

本开发规范的目标不是限制开发效率，而是帮助 OpenMall 在开发初期就建立清晰的协作方式与代码约束。

对于当前这个包含 **用户端 + 商家端 + 后端服务** 的模拟电商项目，统一规范越早建立，后续开发、联调、扩展和维护成本就越低。
