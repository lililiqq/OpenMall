# OpenMall 技术栈说明

> 本文档用于详细说明 OpenMall 项目的技术选型、各组件职责、选型理由以及它们在系统中的协作关系。当前项目为 **Spring Boot + Vue 3** 的前后端分离电商系统，包含 **用户端**、**商家端** 与 **后端服务** 三部分。

---

## 1. 文档目标

本技术栈文档主要用于：

- 明确项目使用的核心开发框架与中间件
- 说明每项技术在系统中的职责边界
- 统一团队对技术选型的理解
- 为开发、部署、扩展和答辩展示提供依据

## 3. 后端核心架构技术

### 3.1 Spring Boot 3.x

Spring Boot 是本项目的后端核心开发框架，负责：

- 提供 Web API 能力
- 管理 Bean 与依赖注入
- 整合 MyBatis-Plus、Redis、RabbitMQ、OSS 等组件
- 管理多环境配置
- 组织模块化单体架构

选择 Spring Boot 3.x 的原因：

- 生态成熟，企业项目使用广泛
- 与 Java 17 配合较好，适合中大型后端项目
- 对 Spring Security、Redis、RabbitMQ 等支持成熟
- 适合作为校招、实习、毕业设计项目的主框架

在 OpenMall 中，Spring Boot 主要负责：

- 用户端与商家端 RESTful API
- 订单、商品、库存、支付等核心业务逻辑
- 中间件整合与配置管理

---

### 3.2 Spring Security + JWT

Spring Security + JWT 是项目的统一安全认证方案。

主要职责：

- 用户端登录认证
- 商家端登录认证
- Token 校验
- 接口访问权限控制
- 用户端与商家端身份隔离

为什么这样选：

- Spring Security 提供完整安全框架能力
- JWT 适合前后端分离架构
- 能避免传统 Session 在多端场景下的扩展限制
- 方便与移动端、小程序、后台管理端统一鉴权方式

在本项目中的典型应用：

- 用户登录后获取 JWT Token
- 商家登录后获取独立身份 Token
- 请求头通过 `Authorization: Bearer <token>` 访问受保护接口
- 通过角色或权限区分用户端接口和商家端接口

---

### 3.3 MyBatis-Plus

MyBatis-Plus 是本项目的持久层开发框架。

主要职责：

- 简化单表 CRUD
- 提供条件构造器
- 提供分页能力
- 减少重复 Mapper 代码

选择理由：

- 比原生 MyBatis 开发效率更高
- 对学习型项目与中小型业务开发友好
- 与 Spring Boot 集成简单
- 能保留 SQL 可控性，同时提升开发效率

在本项目中的适用场景：

- 用户、地址、分类、商品、订单等基础表操作
- 分页查询商品列表与订单列表
- 商家端商品管理与订单管理查询

---

### 3.4 Knife4j

Knife4j 是本项目的接口文档与联调工具。

主要职责：

- 自动生成接口文档
- 提供接口在线调试能力
- 便于前后端联调与测试
- 降低文档维护成本

选择理由：

- 对 Java 后端项目非常友好
- 在国内开发场景使用广泛
- 页面交互体验优于基础 Swagger UI
- 适合在开发期快速验证接口

在本项目中的主要价值：

- 用户端与商家端接口统一可视化展示
- 开发阶段可通过 `/doc.html` 查看接口文档
- 便于答辩演示与测试联调

---

### 3.5 Druid

Druid 是本项目使用的数据库连接池。

主要职责：

- 管理数据库连接
- 提升数据库访问稳定性
- 支持连接池监控与统计
- 支持 SQL 监控与慢 SQL 分析

选择理由：

- 在 Java 项目中使用成熟
- 与 Spring Boot、MySQL 集成方便
- 适合学习连接池监控和数据库性能分析

在 OpenMall 中，Druid 主要用于：

- 管理 MySQL 连接资源
- 配合开发环境查看 SQL 执行情况
- 监控 SQL 执行性能并辅助识别慢查询
- 为商品列表、订单查询等场景提供稳定连接支持

## 4. 数据与中间件技术

### 4.1 MySQL 8.0.34

MySQL 8.0.34 是项目的主数据库。

主要职责：

- 存储用户、商家、店铺、商品、订单、支付等核心业务数据
- 保证交易数据的持久化与一致性
- 支撑订单、库存、支付等强业务约束场景

选择理由：

- 关系型数据库适合电商业务建模
- 与 Spring Boot、MyBatis-Plus、Druid 集成成熟
- 社区资料丰富，适合学习与项目展示

在项目中的数据重点：

- `user` / `merchant` / `shop`
- `product_spu` / `product_sku`
- `cart_item`
- `orders` / `order_item`
- `payment_record`
- `file_record`（可选，用于记录 OSS 文件元信息）

工程注意事项：

- 文档声明的 MySQL 版本应尽量与 Docker 镜像版本、运行环境版本保持一致
- JDBC 驱动版本也应与目标 MySQL 版本保持兼容，避免时区、认证插件、加密连接等问题
- 开发、测试、演示环境建议统一版本标签，降低跨环境差异

---

### 4.2 Redis + Spring Cache

Redis 是项目的高性能缓存组件，Spring Cache 是缓存抽象层。

二者结合后的主要职责：

- 缓存商品详情和商品列表热点数据
- 缓存登录态与验证码
- 实现部分防重复提交与幂等控制
- 统一缓存注解与缓存策略管理

为什么这样组合：

- Redis 性能高，适合读多写少与热点场景
- Spring Cache 可以降低缓存接入成本
- 通过注解方式即可管理常见缓存逻辑

在 OpenMall 中的典型场景：

- 商品详情缓存
- 商品分类缓存
- 首页热门商品缓存
- 用户登录态或 Token 黑名单扩展场景

与 JWT 的协作边界：

- JWT 负责携带用户或商家的身份与权限信息
- Redis 不是 JWT 的必需状态存储，而是用于补充会话控制能力
- 在需要主动注销、强制下线、多端登录限制时，可通过 Redis 维护 Token 黑名单或附加有效期控制

设计原则：

- Redis 只作为缓存或辅助存储，不作为主业务事实源
- 缓存应设置明确过期时间
- 更新商品数据时应同步处理缓存失效
- 认证体系建议采用“JWT 主导 + Redis 辅助”的方式，避免职责混淆

---

### 4.3 RabbitMQ

RabbitMQ 是项目的消息队列组件。

主要职责：

- 异步处理非强实时业务
- 解耦订单与后续处理逻辑
- 承载延迟关闭订单、库存回补等业务流程

选择理由：

- 教学与中小项目场景使用广泛
- 模型清晰，适合学习消息路由和异步解耦
- 与 Spring Boot 集成成熟

在本项目中的典型场景：

- 订单超时关闭
- 取消订单后库存回补
- 支付成功后的异步通知或日志处理
- 商家操作后的异步统计任务

设计原则：

- 先保证同步主流程正确，再引入消息异步化
- 消费者必须考虑幂等性
- 队列、交换机、路由键命名要清晰统一
- 关键业务消息应关注生产者确认机制（Confirm）、消费者确认机制（Ack）与消息持久化配置
- 对订单超时关闭、库存回补等关键链路，建议配套失败重试或死信队列策略，降低消息丢失风险

---

### 4.4 阿里云 OSS

阿里云 OSS 是项目的文件存储方案。

主要职责：

- 存储商品主图与轮播图
- 存储店铺 Logo、店铺装修图片
- 存储富文本详情中的图片资源
- 为前端提供稳定的图片访问地址

选择理由：

- 对电商项目非常贴合
- 适合图片、静态文件集中存储
- 比本地文件系统更适合后续线上部署
- 便于展示项目具备实际云服务集成能力

在 OpenMall 中的建议用法：

- 商家在商品管理端上传图片到 OSS
- 后端返回可访问的文件 URL
- 可在数据库中记录 `objectKey`、文件地址、文件类型、上传人等元信息

环境隔离建议：

- 生产环境使用阿里云 OSS 承载正式文件资源
- 本地开发或测试环境可使用 MinIO 作为对象存储替代方案，以降低开发成本并支持离线调试
- 不同环境应使用不同的 Bucket、端点配置和访问凭据，避免开发数据与生产数据混用

## 5. 前端技术栈

### 5.1 Vue 3

Vue 3 是本项目的前端核心框架。

主要职责：

- 构建用户端页面
- 构建商家端页面
- 管理组件化页面结构
- 配合接口完成前后端分离交互

选择理由：

- 中文生态成熟
- 上手成本适中，组件化开发体验好
- 非常适合作为中后台与电商前端项目框架

在项目中的页面划分：

- 用户端：商品列表、商品详情、购物车、订单页、个人中心
- 商家端：登录页、工作台、商品管理、订单管理、店铺信息管理

### 5.2 Vue 生态补充组件

虽然核心框架为 Vue 3，但项目工程化还会搭配以下常见工具：

- **Vite**：前端构建工具
- **Vue Router**：前端路由管理
- **Pinia**：状态管理
- **Axios**：接口请求
- **Element Plus**：商家端中后台 UI 组件

这些工具在用户端与商家端中承担的角色不同：

- 用户端更偏商品展示与购物流程
- 商家端更偏表单、表格、后台管理交互

## 6. 部署与运维技术

### 6.1 Nginx

Nginx 是项目的 Web 服务器与反向代理组件。

主要职责：

- 部署用户端静态资源
- 部署商家端静态资源
- 反向代理后端 API
- 统一域名与访问入口

在本项目中的典型使用方式：

- `/` 映射用户端前端页面
- `/merchant/` 映射商家端前端页面
- `/api/` 反向代理到 Spring Boot 服务
- `/merchant/api/` 反向代理到商家端接口

### 6.2 Docker

Docker 是项目的容器化部署技术。

主要职责：

- 快速构建开发、测试、演示环境
- 保证不同机器上的运行环境一致
- 简化 MySQL、Redis、RabbitMQ、Nginx 等组件部署

建议使用方式：

- 开发阶段：可通过 Docker 启动 Redis、RabbitMQ、MySQL
- 演示阶段：可通过 Docker / Docker Compose 一键拉起整体环境

### 6.3 Git

Git 是项目的版本控制工具。

主要职责：

- 管理代码版本
- 记录需求与迭代历史
- 支持多人协作开发
- 支持功能分支与缺陷修复分支管理

推荐分支结构：

- `main`：稳定分支
- `dev`：开发分支
- `feature/*`：功能分支
- `fix/*`：修复分支
- `docs/*`：文档分支

## 7. 技术栈协作关系

OpenMall 的技术栈协作链路如下：

1. 用户端和商家端通过 Vue 3 页面发起请求
2. Nginx 统一承接静态资源和接口代理
3. Spring Boot 提供 REST API 与业务逻辑
4. Spring Security + JWT 负责鉴权与权限隔离
5. MyBatis-Plus + Druid 负责数据库访问与连接管理
6. MySQL 存储主业务数据
7. Redis + Spring Cache 提供缓存能力
8. RabbitMQ 负责异步任务处理
9. 阿里云 OSS 提供图片与文件存储能力
10. Knife4j 提供接口文档与调试入口

## 8. 当前技术选型的优势

### 8.1 对学习项目友好

- 技术栈完整但不过度复杂
- 能覆盖后端、前端、数据库、中间件、部署多个知识点
- 非常适合作为作品集展示项目

### 8.2 对电商业务贴合度高

- MySQL 适合订单与库存建模
- Redis 适合热点商品与缓存场景
- RabbitMQ 适合订单超时与库存回补
- OSS 适合商品图片和店铺资源存储

### 8.3 具备扩展空间

- 当前可采用模块化单体架构快速落地
- 后续可逐步增强监控、搜索、营销活动、微服务拆分等能力

对于搜索能力的扩展，需要额外考虑搜索引擎与主数据库之间的数据同步问题。若未来引入 Elasticsearch，常见方向包括：

- 通过 Canal 实现 MySQL Binlog 增量同步
- 通过 Logstash 或其他数据管道做异步同步
- 通过业务事件驱动方式同步搜索索引

这类方案的主要风险在于数据最终一致性、重复消费、乱序或同步失败。对于当前 MVP 阶段，不建议过早承诺复杂搜索链路，而更适合作为后续演进方向。

---

## 9. 当前阶段不重点引入的技术

为了控制项目复杂度，当前阶段不优先引入：

- 微服务注册中心
- 配置中心
- Elasticsearch
- 分布式事务框架
- Kubernetes

原因是本项目当前重点仍然是：

- 用户端交易流程闭环
- 商家端商品与订单管理闭环
- 技术栈清晰、文档齐全、易于展示

其中，Elasticsearch 本身并不是问题的全部，真正的工程复杂度通常来自 MySQL 到 Elasticsearch 的增量同步链路设计。若后续需要引入搜索能力，可进一步研究 Canal、Logstash 等方案，并评估同步时延、幂等处理、数据校验与回放机制。

---

## 10. 总结

OpenMall 当前的技术栈是一套典型的 Java 电商项目技术组合：

- **Spring Boot 3.x** 负责后端核心开发
- **Spring Security + JWT** 负责安全认证
- **MyBatis-Plus + Druid + MySQL 8.0.34** 负责数据访问与持久化
- **Redis + Spring Cache** 负责缓存优化
- **RabbitMQ** 负责异步解耦
- **阿里云 OSS** 负责文件存储
- **Vue 3** 负责用户端与商家端页面构建
- **Nginx + Docker** 负责部署与环境管理
- **Git** 负责版本控制

这套技术栈既能够支撑当前模拟淘宝项目的核心业务，也具备较好的工程展示价值与后续扩展空间。
