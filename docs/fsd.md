# OpenMall 功能规格说明书（FSD）

> 本文档为 OpenMall 项目的功能规格说明书（Functional Specification Document）。
> 目标是用尽可能完整、清晰、可执行的方式描述项目的功能范围、业务规则、交互边界、状态流转、异常场景与非功能要求，便于产品设计、前后端开发、测试联调和后续扩展。

---

## 1. 文档目的

本 FSD 用于明确以下内容：

- 项目整体业务范围
- 用户端与商家端的功能边界
- 核心业务流程与状态流转
- 主要页面 / 接口 / 数据对象之间的关系
- 功能输入、输出、规则、异常与约束
- 后续测试、联调与迭代依据

本文件不替代：

- `docs/API.md`（接口级设计）
- `docs/DB.md`（数据库设计）
- `docs/DESIGN.md`（总体设计）

而是从“功能视角”把这些内容串起来。

## 2. 项目概述

### 2.1 项目名称

- 项目名称：OpenMall
- 项目类型：前后端分离模拟电商平台
- 业务定位：模拟淘宝核心交易流程的学习型 B2C / 类 B2B2C 电商项目

### 2.2 项目目标

OpenMall 的目标不是完整复刻真实淘宝，而是在合理范围内实现一个可运行、可演示、可扩展的电商系统，重点覆盖：

- 用户注册与登录
- 商品浏览与商品详情
- 购物车管理
- 提交订单
- 模拟支付
- 订单查询与取消
- 商家商品管理
- 商家订单履约管理

### 2.3 当前版本边界

当前版本聚焦基础交易闭环，不纳入：

- 优惠券系统
- 秒杀系统
- 推荐系统
- 实时 IM / 客服系统
- 完整搜索引擎系统
- 已支付退款 / 退货 / 售后体系
- 商家入驻审核体系
- 多租户微服务拆分

## 3. 角色定义

### 3.1 普通用户

普通用户是平台消费者，主要职责与能力：

- 注册账号
- 登录系统
- 浏览商品分类与商品列表
- 查看商品详情
- 选择商品 SKU
- 加入购物车
- 管理购物车
- 提交订单
- 模拟支付
- 查询订单列表与详情
- 取消未支付订单
- 管理收货地址

### 3.2 商家

商家是平台的供给侧角色，主要职责与能力：

- 登录商家端系统
- 维护店铺资料
- 新增商品
- 编辑商品
- 上下架商品
- 维护商品 SKU、价格、库存
- 查看订单列表
- 查看订单详情
- 发货并更新履约状态

### 3.3 系统

系统负责：

- 身份认证与权限控制
- 核心数据持久化
- 缓存管理
- 异步处理
- 文件存储
- 订单状态流转控制
- 数据一致性与幂等控制

## 5. 用户端功能规格

---

## 5.1 用户注册

### 5.1.1 功能目标

允许新用户通过用户名、密码、手机号创建平台账号。

### 5.1.2 输入项

- 用户名
- 密码
- 确认密码
- 手机号

### 5.1.3 输出结果

- 注册成功
- 返回用户基础信息

### 5.1.4 业务规则

- 用户名不可重复
- 手机号格式必须合法
- 密码与确认密码必须一致
- 密码不得明文存储

### 5.1.5 异常场景

- 用户名已存在
- 参数缺失
- 手机号格式错误

---

## 5.2 用户登录

### 5.2.1 功能目标

用户通过账号密码登录系统，并获得访问令牌。

### 5.2.2 输入项

- 用户名
- 密码

### 5.2.3 输出结果

- Token
- 用户基础资料

### 5.2.4 业务规则

- 登录成功后返回 JWT Token
- 后续请求通过 `Authorization: Bearer <token>` 访问受保护接口
- Token 失效时返回业务码 `4002`
- 若后续启用刷新机制，可通过 refresh 接口换取新 Token

### 5.2.5 异常场景

- 用户不存在
- 密码错误
- 账号被禁用

---

## 5.3 获取当前用户信息

### 5.3.1 功能目标

获取当前登录用户的基础资料。

### 5.3.2 业务规则

- 必须已登录
- 根据当前 Token 中的用户身份获取资料

---

## 5.4 收货地址管理

### 5.4.1 功能范围

- 获取地址列表
- 新增地址
- 修改地址
- 删除地址

### 5.4.2 地址字段

- 收货人姓名
- 收货手机号
- 省 / 市 / 区
- 详细地址
- 是否默认地址

### 5.4.3 业务规则

- 一个用户可拥有多个地址
- 默认地址最多一个
- 地址只能由本人查看和修改

### 5.4.4 异常场景

- 地址不存在
- 越权访问他人地址
- 手机号格式错误

---

## 5.5 分类浏览

### 5.5.1 功能目标

向用户展示商品分类树，用于导航与筛选。

### 5.5.2 业务规则

- 支持一级分类与二级分类
- 仅返回启用中的分类

---

## 5.6 商品列表

### 5.6.1 功能目标

向用户展示商品分页列表，支持筛选与排序。

### 5.6.2 输入条件

- 页码 `pageNum`
- 每页数量 `pageSize`
- 分类 ID
- 搜索关键字
- 排序规则

### 5.6.3 输出结果

- 分页对象
- 商品基础信息列表

### 5.6.4 商品展示字段

- 商品 ID
- 商品名称
- 主图
- 价格
- 店铺名称
- 销量

### 5.6.5 业务规则

- 仅展示上架商品
- 当前版本搜索能力以基础关键字检索、分类筛选和排序为主
- 当前阶段可基于 MySQL 完成基础商品检索，不将 Elasticsearch 作为当前版本必选能力
- 分页响应统一包含：
  - `pageNum`
  - `pageSize`
  - `total`
  - `list`
  - `hasNextPage`

### 5.6.6 异常场景

- 分类不存在
- 分页参数非法

---

## 5.7 商品详情

### 5.7.1 功能目标

向用户展示商品完整详情和 SKU 规格选择信息。

### 5.7.2 输出内容

- 商品基础信息
- 店铺信息
- 主图和轮播图
- SKU 规格图（若存在）
- 图文详情
- SKU 列表

### 5.7.3 SKU 输出字段

- `skuId`
- `skuName`
- `specJson`
- `price`
- `stock`

### 5.7.4 业务规则

- `specJson` 用于前端渲染规格选择器
- 不同 SKU 可对应不同价格与库存
- 已下架商品不可正常下单

### 5.7.5 异常场景

- 商品不存在
- 商品已下架
- SKU 数据异常

---

## 5.8 加入购物车

### 5.8.1 功能目标

允许用户将指定 SKU 加入购物车。

### 5.8.2 输入项

- `skuId`
- `quantity`

### 5.8.3 核心规则

- 若当前用户购物车中已存在相同 `skuId`，则累加数量
- 若不存在，则新增购物车记录
- 该接口应具备幂等语义

### 5.8.4 业务校验

- 用户必须登录
- SKU 必须存在
- 商品必须可售
- 数量必须大于 0

### 5.8.5 异常场景

- SKU 不存在
- 商品已下架
- 库存不足

---

## 5.9 购物车列表

### 5.9.1 功能目标

展示当前用户购物车中的商品项。

### 5.9.2 输出字段

- 购物车项 ID
- 商品信息
- SKU 信息
- 数量
- 勾选状态
- 单价
- 小计金额

### 5.9.3 业务规则

- 仅返回当前用户购物车数据
- 数据以 MySQL 持久化为准
- 高频场景可结合 Redis 做缓存加速
- 购物车中的价格用于展示与结算前参考，不作为最终成交价承诺
- 用户进入购物车或结算页时，应以服务端返回的最新 SKU 价格为准
- 若价格发生变动，前端应提示用户“价格变化”或“降价”信息

---

## 5.10 修改购物车数量与勾选状态

### 5.10.1 功能目标

修改购物车商品数量与勾选状态。

### 5.10.2 输入项

- 数量
- 勾选状态

### 5.10.3 业务规则

- 数量必须大于 0
- 勾选状态为布尔语义字段
- 购物车项必须属于当前用户

---

## 5.11 删除购物车商品

### 5.11.1 功能目标

从购物车中移除指定商品项。

### 5.11.2 业务规则

- 只能删除自己的购物车项

---

## 5.12 订单预览

### 5.12.1 功能目标

基于购物车勾选商品生成订单预览结果。

### 5.12.2 输入项

- 地址 ID
- 勾选购物车项 ID 列表

### 5.12.3 输出内容

- 店铺分组商品列表
- 商品明细
- 总金额
- 运费
- 实付金额

### 5.12.4 业务规则

- 必须校验地址合法性
- 必须校验商品状态、库存、价格
- 多店铺商品可按店铺分组返回

### 5.12.5 价格与库存结算规则

- 订单预览用于结算前校验与金额重算，不在该阶段锁定库存
- 若购物车展示价与当前 SKU 最新价格不一致，应返回最新价格并提示前端展示价格变更
- 订单预览结果不构成最终成交承诺，提交订单时仍需再次校验价格与库存

---

## 5.13 创建订单

### 5.13.1 功能目标

用户提交订单并生成订单主记录与订单明细。

### 5.13.2 输入项

- 地址 ID
- 购物车项 ID 列表
- 用户备注

### 5.13.3 输出结果

- 订单 ID
- 订单编号
- 订单状态
- 实付金额

### 5.13.4 业务规则

- 下单必须在事务中完成：
  - 校验商品
  - 校验库存
  - 校验最新价格
  - 创建订单
  - 创建明细
  - 锁定库存
  - 清理购物车
- 创建订单时锁定库存，不在加入购物车或订单预览阶段占用库存
- 未支付订单应具备支付有效期，建议默认为 15 分钟；超时后由延迟任务关闭订单并释放锁定库存
- 订单明细应保存商品名称、规格、主图、单价等成交快照信息
- 金额字段统一按货币精度计算
- 状态默认：
  - `order_status = UNPAID`
  - `pay_status = UNPAID`
  - `delivery_status = PENDING_DELIVERY`

### 5.13.5 异常场景

- 库存不足
- 商品已下架
- 地址不存在
- 重复提交订单

---

## 5.15 查询订单详情

### 5.15.1 功能目标

获取指定订单的完整信息。

### 5.15.2 返回内容

- 状态信息
- 收货信息
- 商品明细

- 物流信息（若存在）

### 5.15.3 业务规则

- 只能查询自己的订单

---

## 5.16 取消订单

### 5.16.1 功能目标

用户取消未支付订单。

### 5.16.2 输入项

- 取消原因

### 5.16.3 状态规则

- 仅允许：`UNPAID -> CANCELED`
- 不允许取消已支付或已发货订单
- 系统超时未支付订单应进入 `CLOSED`，不与用户主动取消共用同一语义

### 5.16.4 联动处理

- 更新订单主状态
- 释放锁定库存
- 必要时写订单日志

---

## 5.17 模拟支付

### 5.17.1 功能目标

模拟用户对订单进行支付。

### 5.17.2 输入项

- 支付渠道

### 5.17.3 输出内容

- 订单编号
- 支付状态
- 支付时间

### 5.17.4 状态规则

- `pay_status: UNPAID -> PAID`
- `order_status: UNPAID -> PROCESSING`

### 5.17.5 业务规则

- 支付接口需限制重复支付
- 支付成功后写入支付记录
- 支付前订单必须处于 `UNPAID` 且未被取消或关闭
- 支付成功后确认锁定库存进入有效成交结果，不再执行重复释放

## 5.18 查询支付记录

### 5.18.1 功能目标

查询指定订单的支付信息。

## 5.19 支付成功异步回调（预留）

### 5.19.2 特点

## 7.1 统一响应结构

### 7.1.1 标准结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "timestamp": 1712034000000
}
```

### 7.1.2 字段说明

- `code`：业务状态码
- `message`：业务说明
- `data`：业务数据
- `timestamp`：服务端响应时间戳

---

## 7.2 统一分页结构

```json
{
  "pageNum": 1,
  "pageSize": 10,
  "total": 100,
  "list": [],
  "hasNextPage": true
}
```

## 7.3 业务状态码

建议至少包含：

- `0`：成功
- `4000`：通用业务失败
- `4001`：参数校验失败
- `4002`：未登录或 Token 无效
- `4003`：无权限访问
- `4004`：资源不存在
- `4005`：状态不允许当前操作
- `4006`：库存不足
- `4007`：文件上传失败
- `4008`：OSS 访问异常
- `5000`：系统异常

## 7.4 认证与授权

### 7.4.1 用户端

- 登录获取 JWT Token
- 登录后访问地址、购物车、订单、支付接口

### 7.4.2 商家端

- 登录获取商家 Token
- 接口需校验商家身份与资源归属

### 7.4.3 权限规则

- 用户 Token 不可访问商家端接口
- 商家 Token 不可访问用户私有接口
- 后端不能只信任前端传入的 `merchantId` / `shopId`
- 商家端列表接口必须按当前登录商家所属店铺做数据范围过滤
- 商家端详情、修改、上下架、库存调整、发货等接口必须校验资源归属
- 商家端批量操作也必须逐项校验资源是否属于当前店铺

## 7.5 文件上传能力

### 7.5.1 功能目标

支持商品图片、店铺 Logo、富文本图片上传。

### 7.5.2 业务规则

- 统一由后端接入 OSS
- 上传时校验类型、大小、业务归属
- 返回可访问 URL
- 记录文件元信息
- 建议按业务类型区分文件用途，例如 `PRODUCT_MAIN`、`PRODUCT_ALBUM`、`SKU_IMAGE`、`SHOP_LOGO`、`EDITOR_IMAGE`
- 上传记录应可关联业务主键、上传人身份与 OSS 对象 Key，便于审计和清理

## 7.6 缓存能力

### 7.6.1 使用场景

- 商品详情缓存
- 分类缓存
- 登录态辅助控制
- 购物车短期加速

### 7.6.2 原则

- Redis 不替代 MySQL 主事实源
- 购物车可加速，但最终以数据库结果为准

## 7.7 消息队列能力

### 7.7.1 使用场景

- 订单超时关闭
- 库存回补
- 支付结果异步通知

### 7.7.2 原则

- 消息消费需考虑幂等
- 关键消息关注 Confirm / Ack / 持久化
- 对未支付订单建议通过延迟任务执行超时关闭，并在处理时释放锁定库存

## 8. 状态机规格

---

## 8.1 订单主状态

- `UNPAID`
- `PROCESSING`
- `COMPLETED`
- `CANCELED`
- `CLOSED`

## 8.2 支付状态

- `UNPAID`
- `PAID`
- `FAILED`

## 8.3 履约状态

- `PENDING_DELIVERY`
- `SHIPPED`
- `RECEIVED`

## 8.4 状态流转规则

- 创建订单：
  - `order_status = UNPAID`
  - `pay_status = UNPAID`
  - `delivery_status = PENDING_DELIVERY`
- 支付成功：
  - `pay_status: UNPAID -> PAID`
  - `order_status: UNPAID -> PROCESSING`
- 发货成功：
  - `delivery_status: PENDING_DELIVERY -> SHIPPED`
- 用户取消：
  - `order_status: UNPAID -> CANCELED`
- 系统超时关闭：
  - `order_status: UNPAID -> CLOSED`
- 用户取消或系统关闭后，均应释放锁定库存
- 已取消订单不可进入发货流程
- 已关闭订单不可继续支付

## 9. 异常与边界场景

### 9.1 用户端常见异常

- 用户未登录
- Token 失效
- 商品不存在
- 商品已下架
- SKU 不存在
- 库存不足
- 商品价格发生变化
- 订单状态异常
- 订单已超时关闭
- 地址不存在

### 9.2 商家端常见异常

- 非商家身份访问商家接口
- 越权访问其他店铺资源
- 商品不属于当前店铺
- 订单不属于当前店铺
- 已取消订单发货

### 9.3 公共异常

- 参数校验失败
- 文件上传失败
- OSS 访问异常
- 系统异常
