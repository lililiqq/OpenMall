# OpenMall OpenMall项目设计文档

## 1. 项目概述

### 1.1 项目背景

OpenMall 是一个参考淘宝核心业务流程设计的电商练手项目，目标是实现一个具备基础交易闭环的前后端分离系统。项目聚焦于常见电商场景，如商品浏览、购物车、下单、支付模拟、订单管理与后台管理，适合作为学习项目、毕业设计或个人作品集项目。

### 1.2 项目目标

本项目的目标包括：

- 熟悉 Spring Boot 后端项目的分层设计与模块划分
- 熟悉 Vue3 前端项目的工程化开发方式
- 掌握 MySQL 在电商场景下的数据建模思路
- 理解 Redis 在缓存、登录态、幂等控制中的作用
- 理解 RabbitMQ 在异步解耦、延迟任务中的应用场景
- 完成一个可运行、可演示、可扩展的基础电商系统

### 1.3 项目定位

这是一个“模拟淘宝”的学习型项目，而不是一个生产级的大型电商平台。项目会优先保证：

- 核心业务流程完整
- 系统结构清晰
- 文档可读性强
- 便于后续迭代扩展

不追求首版就覆盖复杂营销、推荐、搜索引擎、分布式微服务等重型能力。

## 2. 需求分析

### 2.1 用户角色

系统初期包含两个核心角色：

#### 2.1.1 普通用户

普通用户可以完成以下操作：

- 注册、登录、退出登录
- 浏览商品分类
- 查看商品列表和商品详情
- 选择商品规格（SKU）
- 将商品加入购物车
- 管理购物车中的商品
- 提交订单
- 进行支付模拟
- 查询订单列表与订单详情
- 取消未支付订单
- 管理收货地址

#### 2.1.2 管理员

管理员可以完成以下操作：

- 管理商品分类
- 新增、编辑、上下架商品
- 维护商品库存
- 查询订单信息
- 更新订单状态

### 2.2 核心业务需求

#### 2.2.1 用户与认证

- 支持用户注册与登录
- 支持基于 Token 的身份认证
- 支持获取当前登录用户信息

#### 2.2.2 商品系统

- 支持商品分类管理
- 支持商品分页查询
- 支持商品详情查询
- 支持商品 SPU 与 SKU 管理
- 支持商品上下架状态控制

#### 2.2.3 购物车系统

- 支持商品加入购物车
- 支持购物车商品数量修改
- 支持删除购物车商品
- 支持勾选部分商品进行结算

#### 2.2.4 订单系统

- 支持订单预览
- 支持创建订单
- 支持订单列表与详情查询
- 支持取消未支付订单
- 支持订单状态流转

#### 2.2.5 支付系统

- 支持模拟支付
- 支持支付结果写回订单状态
- 支持查询支付记录

#### 2.2.6 后台管理系统

- 支持商品管理
- 支持库存管理
- 支持订单管理

### 2.3 非功能需求

#### 2.3.1 可维护性

- 后端采用清晰的模块分层结构
- 核心业务命名统一
- 文档齐全，便于后续维护和扩展

#### 2.3.2 可扩展性

- 支持后续新增评价、优惠券、秒杀等模块
- 支持后续引入搜索引擎、对象存储、微服务拆分等能力

#### 2.3.3 性能要求

- 商品详情与商品列表支持 Redis 缓存优化
- 异步任务使用 RabbitMQ 解耦，降低同步链路压力

#### 2.3.4 安全性

- 接口应具备基础鉴权能力
- 管理端与用户端接口权限隔离
- 关键接口应考虑防重复提交与幂等控制

## 3. 技术栈设计

### 3.1 后端技术栈

- Java 17
- Spring Boot 3.x
- Spring Web
- JWT
- MyBatis-Plus
- Maven

### 3.2 前端技术栈

- Vue 3
- Vite
- Vue Router
- Pinia
- Axios
- Element Plus

### 3.3 中间件与基础设施

- MySQL 8.x：核心业务数据存储
- Redis：缓存、登录态、幂等控制
- RabbitMQ：异步消息处理

### 3.4 技术选型说明

#### 3.4.1 为什么使用 Spring Boot

- 生态完善，适合快速搭建后台系统
- 与 MySQL、Redis、RabbitMQ 集成成熟
- 适合作为 Java 后端学习与面试项目基础框架

#### 3.4.2 为什么使用 Vue3

- 生态成熟，配合 Vite 开发体验较好
- 组件化开发方便构建电商页面
- 适合搭配 RESTful API 完成前后端分离项目

#### 3.4.3 为什么使用 MySQL

- 关系型数据库适合订单、库存、用户等强一致业务
- 电商业务实体关系较明确，适合结构化建模

#### 3.4.4 为什么使用 Redis

- 适合承载高频读取数据的缓存场景
- 可用于登录态缓存、验证码、热点商品数据缓存
- 可用于简单幂等 Token 与限流控制

#### 3.4.5 为什么使用 RabbitMQ

- 适合订单超时关闭、库存回补、异步通知等场景
- 能降低同步调用链路耦合度
- 易于体现项目的异步架构能力

## 4. 系统架构设计

### 4.1 总体架构

本项目采用“前后端分离 + 模块化单体”架构。

#### 架构分层

- 前端层：Vue3 用户端 / 管理端
- 接口层：Spring Boot REST API
- 业务层：商品、购物车、订单、支付等领域服务
- 数据层：MySQL + Redis
- 消息层：RabbitMQ

### 4.2 当前阶段架构原则

- 首版以模块化单体为主，不一开始拆分微服务
- 保证业务闭环优先于技术复杂度
- 各模块边界清晰，后续具备服务拆分基础

### 4.3 推荐模块划分

- `common`：通用返回、异常处理、工具类
- `user`：用户与认证模块
- `address`：收货地址模块
- `category`：分类模块
- `product`：商品模块
- `cart`：购物车模块
- `order`：订单模块
- `inventory`：库存模块
- `payment`：支付模块
- `admin`：后台管理模块

### 4.4 分层建议

每个业务模块内部建议采用以下分层：

- `controller`：接口层
- `service`：业务层
- `domain` 或 `entity`：核心领域对象
- `repository` / `mapper`：数据访问层
- `dto` / `vo`：请求与响应对象

## 5. 数据库设计思路

### 5.1 设计原则

- 以业务闭环为中心进行建模
- 使用 SPU + SKU 表达商品与规格
- 订单信息采用主表 + 明细表设计
- 核心交易数据优先存储于 MySQL
- 避免将 Redis 作为主数据源

### 5.2 核心数据实体

建议首版包含以下核心表：

- `user`：用户表
- `user_address`：用户地址表
- `category`：商品分类表
- `product_spu`：商品 SPU 表
- `product_sku`：商品 SKU 表
- `cart_item`：购物车项表
- `orders`：订单主表
- `order_item`：订单明细表
- `payment_record`：支付记录表

### 5.3 表设计思路说明

#### 5.3.1 用户表 `user`

用于存储用户的基础信息，如用户名、手机号、密码摘要、状态、创建时间等。

#### 5.3.2 用户地址表 `user_address`

用于存储收货地址信息，支持一个用户维护多个地址，并可设置默认地址。

#### 5.3.3 分类表 `category`

用于维护商品分类，可支持一级分类和二级分类。

#### 5.3.4 商品 SPU 表 `product_spu`

用于描述商品的公共信息，例如商品名称、品牌、简介、主图、所属分类、上下架状态等。

#### 5.3.5 商品 SKU 表 `product_sku`

用于描述商品具体规格信息，例如颜色、尺寸、价格、库存、SKU 编码等。

#### 5.3.6 购物车项表 `cart_item`

用于存储用户加入购物车的商品记录，包括用户 ID、SKU ID、数量、勾选状态等。

#### 5.3.7 订单主表 `orders`

用于记录订单整体信息，包括订单编号、用户 ID、总金额、订单状态、支付状态、收货信息快照、创建时间等。

#### 5.3.8 订单明细表 `order_item`

用于记录订单中的具体商品信息。建议保留商品名称、单价、规格文本等快照字段，避免商品后续修改影响历史订单展示。

#### 5.3.9 支付记录表 `payment_record`

用于记录支付请求、支付状态、支付流水号、支付时间等信息。

### 5.4 数据一致性思路

- 下单时需校验商品状态、库存与价格
- 创建订单与扣减库存应置于同一事务边界中
- 订单取消或支付超时后，应支持库存回补
- 支付成功后再更新订单状态，避免脏数据

## 6. Redis 与 RabbitMQ 使用思路

### 6.1 Redis 使用场景

建议在项目中逐步引入 Redis，优先用于以下场景：

- 登录态缓存
- 验证码缓存
- 热门商品详情缓存
- 商品列表缓存
- 防重复提交 Token

### 6.2 Redis 设计原则

- Redis 作为辅助缓存层，不作为主业务事实源
- 商品缓存可采用 Cache Aside 模式
- 注意缓存失效策略与数据一致性控制

### 6.3 RabbitMQ 使用场景

RabbitMQ 更适合在二期或增强版中使用，典型场景包括：

- 订单超时关闭
- 取消订单后的库存回补
- 支付成功后的异步通知
- 销量统计、行为日志异步处理

### 6.4 RabbitMQ 设计原则

- 优先让核心流程先跑通，再引入异步化优化
- 异步处理的前提是业务边界清晰、状态流转明确
- 消费逻辑要考虑重复消费与幂等控制

## 7. 核心业务流程设计

### 7.1 商品浏览流程

1. 用户进入首页
2. 查询商品分类与商品列表
3. 根据分类或关键词筛选商品
4. 查看商品详情页
5. 选择 SKU 与购买数量

### 7.2 加入购物车流程

1. 用户选择商品 SKU
2. 前端调用加入购物车接口
3. 后端校验商品状态与库存
4. 若购物车中已存在同一 SKU，则累加数量
5. 返回最新购物车数据

### 7.3 下单流程

1. 用户勾选购物车商品并进入结算页
2. 系统读取商品、库存、价格与收货地址信息
3. 用户提交订单
4. 后端校验库存、计算订单金额
5. 创建订单与订单明细
6. 扣减库存
7. 返回订单号与订单信息

### 7.4 支付模拟流程

1. 用户点击支付
2. 前端调用模拟支付接口
3. 后端生成支付记录
4. 更新订单为已支付状态
5. 返回支付成功结果

### 7.5 订单取消流程

1. 用户取消未支付订单
2. 系统校验订单状态
3. 更新订单状态为已取消
4. 回补库存

## 8. 核心 API 清单

### 8.1 认证与用户模块

- `POST /api/v1/auth/register`：用户注册
- `POST /api/v1/auth/login`：用户登录
- `GET /api/v1/users/me`：获取当前用户信息

### 8.2 地址模块

- `GET /api/v1/addresses`：获取地址列表
- `POST /api/v1/addresses`：新增地址
- `PUT /api/v1/addresses/{id}`：修改地址
- `DELETE /api/v1/addresses/{id}`：删除地址

### 8.3 商品模块

- `GET /api/v1/categories`：获取商品分类
- `GET /api/v1/products`：分页查询商品列表
- `GET /api/v1/products/{id}`：查询商品详情

### 8.4 购物车模块

- `GET /api/v1/cart/items`：获取购物车商品
- `POST /api/v1/cart/items`：加入购物车
- `PUT /api/v1/cart/items/{id}`：修改购物车商品数量
- `DELETE /api/v1/cart/items/{id}`：删除购物车商品

### 8.5 订单模块

- `POST /api/v1/orders/preview`：订单预览
- `POST /api/v1/orders`：创建订单
- `GET /api/v1/orders`：查询订单列表
- `GET /api/v1/orders/{id}`：查询订单详情
- `POST /api/v1/orders/{id}/cancel`：取消订单

### 8.6 支付模块

- `POST /api/v1/payments/{orderNo}/mock-pay`：模拟支付
- `GET /api/v1/payments/{orderNo}`：查询支付记录

### 8.7 管理端模块

- `GET /admin/api/v1/products`：后台商品列表
- `POST /admin/api/v1/products`：新增商品
- `PUT /admin/api/v1/products/{id}`：编辑商品
- `PUT /admin/api/v1/products/{id}/status`：商品上下架
- `PUT /admin/api/v1/skus/{id}/stock`：修改库存
- `GET /admin/api/v1/orders`：后台订单查询
- `PUT /admin/api/v1/orders/{id}/delivery-status`：更新订单状态

## 11. 风险与设计取舍

### 11.1 为什么不一开始做微服务

- 微服务会显著增加网关、注册中心、配置中心、链路追踪等复杂度
- 对学习型项目而言，先完成完整业务闭环更重要
- 模块化单体更适合首版快速落地

### 11.2 为什么 Redis 和 RabbitMQ 不应过度使用

- 如果核心流程尚未跑通，过早引入缓存和消息队列会增加排查难度
- Redis 与 RabbitMQ 应服务于明确问题，而不是为了“技术栈丰富”而引入

### 11.3 当前版本的设计重点

- 业务闭环完整
- 模块职责明确
- 数据模型合理
- 接口边界清晰
- 支持后续平滑迭代

## 13. 总结

OpenMall 的设计重点不是无限接近真实淘宝，而是在合理的项目规模下实现一套结构清晰、流程完整、技术栈明确的模拟电商系统。

在当前阶段，推荐以 Spring Boot + Vue3 + MySQL 为核心，以 Redis 和 RabbitMQ 作为增强能力，先实现商品、购物车、订单、支付模拟与后台管理这些基础模块。这样既能保证项目具备展示价值，也能为后续扩展打下较好的结构基础。
